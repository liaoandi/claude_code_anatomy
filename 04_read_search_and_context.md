# 第 4 章 读文件、搜文件、看上下文

## 为什么读文件不直接用 cat

最简单的做法是：让 agent 自己跑 `cat`、`grep`、`find`。这完全可行，但 Claude Code 没有这么做。

原因是：直接用 shell 命令读文件，系统对这件事没有任何感知。不知道读了多少 token、不知道这个文件有没有超大、不知道路径是不是安全的，也没办法针对不同文件类型做特殊处理。出了问题，什么日志都没有。

所以 Claude Code 把读文件做成了专门的工具，处理了一批 shell 命令处理不了的事情。

**路径安全**：`/dev/zero` 这样的特殊文件，读了就会无限阻塞进程，需要在读之前就识别并拦截。

**Token 预算**：大文件不能整块塞进上下文，工具会根据当前 token 预算决定读多少，或者建议用 offset/limit 分段读。

**文件类型处理**：PDF、Jupyter notebook、图片——这些不能像普通文本一样处理，需要走各自的解析逻辑才能变成模型能理解的内容。

**读取事件通知**：读文件这个动作会通知给其他服务，比如 context 预算追踪，这样系统能知道"刚才读了什么，花了多少 token"。

## 搜索：两个工具，两种用法

Claude Code 有两个搜索工具，用途不同。

GrepTool 是"找内容"：知道要找什么，不知道在哪。底层用 ripgrep，速度快，自动尊重 `.gitignore`，不会把 `node_modules` 里的东西搜出来。结果会做截断，防止海量匹配把上下文打爆。

GlobTool 是"找文件"：知道文件结构的大概样子，要列出候选文件。支持完整的 glob 语法（`**/*.ts`、`src/**/index.*`），结果按修改时间排序，让模型优先关注最近改动的文件。

两个工具分开，是因为它们需要的输入不同，权限粒度也不同——知道"某个目录下有哪些文件"和"知道这些文件的内容"，是两种不同级别的访问。

## 三个工具配合使用的典型模式

实际使用中，这三个工具经常配合：先用 GlobTool 找到相关文件列表，再用 GrepTool 确认哪些文件里有目标内容，最后用 FileReadTool 读取具体内容。

这个分层查找的过程减少了不必要的 token 消耗——不需要把每个可能相关的文件都读进来，而是先缩小范围再精读。
