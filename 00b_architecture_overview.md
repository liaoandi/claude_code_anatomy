# 架构总览

在进入各章之前，先把整体结构看清楚。

## 系统是怎么分层的

Claude Code 从外到内大致可以分成六层：

```
入口层     cli.tsx 做模式分流，决定当前是交互 REPL、后台 daemon，还是 bridge
命令层     slash command 注册表，管理所有 /xxx 命令的存在和可见性
工具层     实际干活的地方：读文件、改文件、跑命令、联网、调 agent
权限层     横切所有工具，每次操作前问一句"用户允许吗"
消息层     把所有工具输出、状态变化、错误统一变成消息，进入会话流
任务层     长时间执行的统一容器，shell 任务、agent 任务都在这里管理
状态层     memory、session、context、config——系统的记忆和配置
```

这不是严格的调用栈，而是理解系统的一个框架。真实代码里这些层会交织，但每层的职责相对清楚。

## 三个贯穿全系统的判断

**能力做成工具，不留给 shell 临时解决。** 读文件、搜索、改文件、联网，每件事都有专门的工具。这些工具不只是对系统调用的包装，每一个都带权限检查、展示逻辑和审计记录。代价是多了一层，收益是每种能力都可控、可见、可审计。

**工具自带展示方式。** 每个工具不只定义"做什么"，还定义"怎么显示"。工具调用的过程和结果都会进消息流，会话可以回放，状态可以审计。这让所有工具复用同一套展示容器，而不是各自发明 UI。

**扩展不破坏内聚。** MCP 在运行时动态注入工具，插件和 skill 把 markdown 编译成命令，feature flag 控制哪些能力在哪个版本可见。不管哪种扩展方式，外部能力都必须走同一套权限和消息机制，没有绕过系统的后门。
